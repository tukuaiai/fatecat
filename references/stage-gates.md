# FateCat Skill 阶段门禁

## 使用方法

- 每推进到下一阶段前，先检查上一阶段的退出条件。
- 若退出条件不满足，不要靠口头承诺硬推进。

## 阶段门禁

### 需求 -> 原型

- 必须具备：
  - 目标用户或调用方
  - 输入和输出
  - 成功标准
  - 主要风险
- 禁止跳过：
  - 不允许在目标和验收都模糊的情况下直接大改代码

### 原型 -> 迭代

- 必须具备：
  - 至少一条成功跑通的命令或最小流程
  - 最好保留一次 `bash scripts/preflight.sh --mode pure --bootstrap --smoke --output-file ... --pretty` 的证据
  - 失败样本或边界样本
  - 当前缺口列表
- 禁止跳过：
  - 原型没证明可行，就不要开始大规模清理结构

### 迭代 -> 成熟方案重构

- 必须具备：
  - 当前实现的痛点证据
  - 可替换候选方案
  - 兼容性与回滚考虑
- 禁止跳过：
  - 不能只因为“看着不爽”就重构

### 成熟方案重构 -> 生产优化

- 必须具备：
  - 替换后的收益说明
  - 对性能、稳定性、维护成本的影响判断
  - 关键回归点
- 禁止跳过：
  - 没有回归路径时，不进入生产优化

### 生产优化 -> 运维加固

- 必须具备：
  - 启动、健康检查、失败恢复入口
  - 配置边界
  - 关键运行依赖
  - 至少一条标准预检命令，供后续 agent 直接复用
- 禁止跳过：
  - 缺少可观察入口时，不宣称“生产可用”

### 运维加固 -> 退役

- 必须具备：
  - 当前服务状态
  - 数据归档策略
  - 下游影响说明
  - 替代方案或终止方案
- 禁止跳过：
  - 不允许直接删仓、删数据、删脚本而不留收口说明

## 完成定义

- “已完成”不等于“代码提交了”。
- “已完成”意味着该阶段产物存在、可读、可验证，并能支撑下一阶段的决策。
