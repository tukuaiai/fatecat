# 当前目录结构

## 目录重构结论

本仓库已经完成从旧式 `libs/ + docs/ + deploy/` 结构向 `assets/ + runtime/ + services/` 结构的收敛。

核心原因：

1. `libs/` 把静态资产、外部仓库、数据库实库混在一起，边界模糊。
2. 顶层 `docs/`、`deploy/` 与 `assets/` 并列，职责分散。
3. 路径仍残留外部项目影子，导致仓库自洽性差。

## 当前推荐结构

```text
fatecat/
├── assets/
│   ├── config/        # 配置模板与配置文件
│   ├── data/          # 静态数据
│   ├── database/      # schema
│   ├── deploy/        # 打包与部署脚本
│   ├── docs/          # 文档
│   ├── fate/          # 输出字段 profile
│   └── vendor/        # 外部成熟仓库
├── runtime/
│   └── database/      # SQLite 实库
├── services/
│   ├── fate_core/     # 纯命理分析内核
│   └── telegram/      # Bot / API 交付层
├── scripts/
└── tests/
```

## 合并映射关系

| 旧目录 | 新目录 | 说明 |
|--------|--------|------|
| `docs/` | `assets/docs/` | 文档归档到资产层 |
| `deploy/` | `assets/deploy/` | 部署脚本归档到资产层 |
| `libs/data/` | `assets/data/` | 静态数据归档到资产层 |
| `libs/database/schema` | `assets/database/` | schema 归档到资产层 |
| `libs/database/*.db` | `runtime/database/` | 实库归入运行态 |
| `libs/external/` | `assets/vendor/` | 外部依赖归档到资产层 |

## 为什么这样更合理

- `assets/` 负责“不会因为运行而变化的东西”。
- `runtime/` 负责“程序一跑就会变化的东西”。
- `services/` 负责“代码与业务能力”。

这样处理后：

1. 路径语义清晰。
2. 配置、数据、文档、部署脚本集中。
3. 纯分析模块与交付模块边界更稳。
4. 后续继续拆 `fate_core` 不会再被旧目录结构牵制。
